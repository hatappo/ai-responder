# Scout Message Analyzer

転職プラットフォームのスカウトメッセージを自動取得し、AI 生成の可能性を判定するスキル。

## 前提条件

このスキルはブラウザ操作が必要なため、以下のいずれかが利用可能であること：

- **Playwright MCP サーバー** — Claude Code の MCP 設定で Playwright が有効化されている
- **Browser Use CLI** — `browser-use` コマンドが利用可能

どちらも利用できない場合は、ユーザーにセットアップを案内する。

## ワークフロー

### Step 1: サービス URL の確認とログイン状態チェック

ユーザーから転職プラットフォームの URL を受け取ったら：

1. ブラウザでそのURLを開く
2. ログイン状態を確認する
3. ログインしていない場合は、ユーザーに手動ログインを依頼する
   - 「ログインページが表示されています。ブラウザでログインしてください。完了したら教えてください。」

ログイン情報を自動入力したり、パスワードを尋ねたりしてはいけない。認証情報の取り扱いはユーザー自身に委ねる。

### Step 2: インボックスの特定と未読メッセージの取得

ログイン後、以下の手順でスカウトメッセージのインボックスを探す：

1. ページ内のナビゲーション要素を確認し、「メッセージ」「スカウト」「受信箱」「inbox」などのリンクを探す
2. 該当するリンクに遷移する
3. 未読メッセージの一覧を取得する

取得する情報（可能な範囲で）：
- 送信者名（企業名・エージェント名）
- 受信日時
- 件名またはメッセージのプレビューテキスト
- メッセージ本文（個別ページに遷移して取得）

**重要**: サイトの構造はプラットフォームによって異なる。要素が見つからない場合は、ページのスクリーンショットを撮影してユーザーに確認を求める。推測で操作を進めない。

### Step 3: 未読メッセージの既読化

各メッセージを開いた時点で既読になるプラットフォームが多いが、明示的な「既読にする」操作が必要な場合はそれも実行する。

取得したメッセージの数と、既読化の状況をユーザーに報告する。

### Step 4: AI 生成判定

各メッセージ本文について、以下の観点で AI 生成の可能性を 0〜100 のスコアで評価する：

#### 判定基準

**AI 生成の可能性が高い特徴（スコアが上がる要因）：**
- 定型的な挨拶・締めくくりのパターン
- 候補者のプロフィールに対する具体的な言及がない、または表面的
- 過度に丁寧で均一なトーン
- 「ご経験を拝見し」「貴殿のご経歴に大変興味を持ち」などの汎用的フレーズの多用
- 企業や案件の具体的な説明が薄い
- 文章構造が画一的（導入→褒め→案件紹介→クロージング）
- 同一送信者からの複数メッセージが類似した構造を持つ

**人間が書いた可能性が高い特徴（スコアが下がる要因）：**
- 候補者の具体的なプロジェクトやスキルへの言及
- 個人的なエピソードや感情表現
- 文体の揺れや口語的表現
- 特定のポジションに対する詳細な説明
- 送信者個人の視点からのコメント

#### スコアの目安
- **0-20**: ほぼ確実に人間が作成
- **21-50**: 人間が作成した可能性が高いが、テンプレートの部分利用あり
- **51-75**: AI 生成の可能性あり（テンプレートベースまたは AI アシスト）
- **76-100**: AI 生成の可能性が高い

### Step 5: 結果の出力

以下のフォーマットで一覧を出力する：

```

取得日時: YYYY-MM-DD HH:MM
プラットフォーム: [サービス名]
未読メッセージ数: N 件（すべて既読に更新済み）

|---|--------|----------|-----------------|-----------|------|

### 詳細分析

**判定根拠:**
- 汎用的なフレーズ「ご経験を拝見し〜」で開始
- プロフィールの具体的なスキルへの言及なし

```

### Step 6: 次のアクションの案内

分析結果を出力したあと、ユーザーに次のアクションを提案する：

- 「特定のメッセージに返信しますか？番号を教えてください。」
- 「AI スコアが高いメッセージをまとめて無視/アーカイブしますか？」
- 「気になるメッセージの全文を確認しますか？」

ここから先はユーザーとの対話で進める。自動で返信や削除を行わない。

## 注意事項

- **認証情報**: ユーザーのログイン情報を記憶したり、平文で保存したりしない
- **操作の確認**: 既読化以外の変更操作（返信・削除・アーカイブ）は必ずユーザーの明示的な承認を得てから実行する
- **レート制限**: ページ遷移の間に適切な待機時間を設ける。サイトに負荷をかけない
- **エラー処理**: ブラウザ操作が失敗した場合はスクリーンショットを撮ってユーザーに状況を共有する
- **プラットフォーム対応**: サイト固有のセレクタやナビゲーションパスはハードコードせず、ページの構造を動的に解析して対応する。初回利用時に操作手順を学習し、ユーザーの指示に基づいて調整する
